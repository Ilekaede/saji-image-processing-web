---
description: Issue番号を受け取り、背景を読み取ってウェブ調査を行い、結果をIssueコメントとして投稿する
argument-hint: <Issue番号> (例: 118)
---

## 前提情報

- Issue 番号は `$ARGUMENTS` から取得する（例: `/research 120` → 番号は `120`）
- **不明・判断不能な場合は妄想で補完しない**。不明点を Issue コメントに列挙して止まり、ユーザーからの情報を待つ
- 調査観点は Issue 本文から読み取る（固定しない）

---

## Step 1: Issue 読み取り

`gh issue view {番号}` で Issue 本文を取得し、以下を把握する:

- **背景・目的**: 開発者がこの Issue を立てた理由
- **対象領域**: 調査すべき技術分野・コンテキスト
- **技術キーワード**: 調査対象となる技術用語・テーマを Issue 本文から抽出する

抽出結果が以下のいずれかに該当する場合は **Step 1a** へ:

- キーワードが 1 件も抽出できない
- 背景・目的が不明確で調査の方向性が定まらない

### Step 1a: 不明時の処理

不明点を `gh issue comment {番号} --body "..."` で投稿して、ここで止まる。

```
## 調査開始前の確認事項（#<Issue番号>）

調査を開始する前に、以下の点を確認させてください。

- （不明点1）
- （不明点2）

ご確認後に情報をいただければ調査を再開します。
```

---

## Step 2: ウェブ調査

抽出したキーワードごとに **fresh エージェント（subagent_type: claude）を1つのメッセージ内で並列起動**し、調査を委ねる。
`fork` は会話コンテキスト全体を引き継ぐため使わないこと。

各エージェントへの指示（キーワードごとに個別に渡す）:

---

**調査エージェントへの指示:**

あなたは技術調査担当です。以下の条件で調査を行ってください。

**調査キーワード**: {キーワード}

**Issue の背景**:
{Step 1 で読み取った背景・目的}

**調査の観点**:

- 対象読者: そのキーワードの技術を実際に実装しているが上手くいかない経験を持つ学生・エンジニア
- 優先する情報: **実運用上の課題とその対応方法**
- 除外する情報: 詳細な理論・数式・アルゴリズムの説明（簡単な導入は可）
- 複数の資料（公式ドキュメント・技術ブログ・Q&Aサイトなど）を参照してまとめること

**出力形式**（Markdown）:

### {キーワード}

#### よくある課題と対応方法

- **課題名**: 現象・原因・対処法の順で説明する

#### 参考資料

- タイトル: URL

不明な点・調査できなかった点は末尾に「不明: 〜」と記載すること。補完・推測での記述は禁止。

---

## Step 3: 情報の取捨選択と統合

各エージェントから返ってきた調査結果を統合し、以下の基準で内容を整理する:

**残す情報**:

- 実運用上の課題と対応方法
- Issue の背景・目的と関連する情報

**除外する情報**:

- 詳細な理論・数式
- Issue の背景と無関係な情報
- エージェントが「不明」と記載した内容（後述の不明処理へ）

エージェントの出力に「不明: 〜」が含まれている場合は **Step 3a** へ。

### Step 3a: 不明点が残った場合

Step 1a と同じ形式で Issue コメントに不明点を列挙して止まり、ユーザーの追加情報を待つ。

---

## Step 4: Issue コメント投稿

`gh issue comment {番号} --body "..."` で以下の形式のコメントを投稿する:

```
## 調査結果（#<Issue番号>）

### <キーワード1>

#### よくある課題と対応方法

- **<課題名>**: <現象>。<原因>。対処法: <対処法>。

### <キーワード2>

...

---

内容を確認し、記事作成を進めてよい場合はその旨を伝えてください。
問題があれば修正点を教えてください。
```

---

## Step 5: ユーザー確認・案内

ユーザーから記事作成を進める許可が出たら（「OK」「進めて」「問題ない」など文脈で判断する）、以下を案内して終了する:

```
記事執筆を開始するには以下を実行してください:

/edit-article <記事ID>
```

記事 ID が Issue 本文に記載されている場合はその値を使う。記載がない場合は「記事 ID をご確認のうえ実行してください」と伝える。
`/edit-article` の自動呼び出しはしない。
