## 前提情報

- Issue 番号は `$ARGUMENTS` から取得する（例: `/research 120` → 番号は `120`）
- リポジトリは `Ilekaede/saji-image-processing-web`（`gh` コマンドには必ず `--repo` を付ける）
- **不明・判断不能な場合は妄想で補完しない**。不明点を Issue コメントに列挙して止まり、ユーザーからの情報を待つ
- 調査観点は Issue 本文から読み取る（固定しない）

---

## Step 1: Issue 読み取り

`gh issue view {番号} --repo Ilekaede/saji-image-processing-web` で Issue 本文を取得し、以下を把握する:

- **背景・目的**: 開発者がこの Issue を立てた理由
- **対象領域**: 調査すべき技術分野・コンテキスト
- **技術キーワード**: 調査対象となる技術用語・テーマを Issue 本文から抽出する（最大3件）

抽出結果が以下のいずれかに該当する場合は **Step 1a** へ:

- キーワードが 1 件も抽出できない
- 背景・目的が不明確で調査の方向性が定まらない

### Step 1a: 不明時の処理

不明点を `gh issue comment {番号} --repo Ilekaede/saji-image-processing-web --body "..."` で投稿して、ここで止まる。

ユーザーから追加情報を受け取ったら、`/research {番号}` を再実行して Step 1 から再開する。

```
## 調査開始前の確認事項（#<Issue番号>）

調査を開始する前に、以下の点を確認させてください。

- （不明点1）
- （不明点2）

ご確認後に情報をいただければ `/research <Issue番号>` を再実行して調査を開始します。
```

---

## Step 2: ウェブ調査

以下の手順でエージェントへ渡すプロンプトを構築する:

1. `{キーワード}` を Step 1 で抽出した実際のキーワードに置換する
2. `{Step 1 で読み取った背景・目的}` を Step 1 で取得した実際の内容に置換する
3. 置換済みのプロンプトを使い、キーワードごとに **fresh エージェント（subagent_type: claude）を1つのメッセージ内で並列起動**する（最大3エージェント同時起動。キーワードが4件以上ある場合は重要度の高い3件を選ぶ）

`fork` は会話コンテキスト全体を引き継ぐため使わないこと。

各エージェントへの指示（キーワードごとに個別に渡す）:

---

**調査エージェントへの指示:**

まず `ToolSearch` ツールで `select:WebSearch,WebFetch` を呼び出してスキーマをロードしてから調査を開始すること。

あなたは技術調査担当です。以下の条件で調査を行ってください。

**調査キーワード**: {キーワード}

**Issue の背景**:
{Step 1 で読み取った背景・目的}

**調査の観点**:

- 対象読者: そのキーワードの技術を実際に実装しているが上手くいかない経験を持つ学生・エンジニア
- 優先する情報: **実運用上の課題とその対応方法**
- 除外する情報: 詳細な理論・数式・アルゴリズムの説明（簡単な導入は可）
- 複数の資料（公式ドキュメント・技術ブログ・Q&Aサイトなど）を参照してまとめること

**出力形式**（Markdown）:

### {キーワード}

#### よくある課題と対応方法

- **課題名**: 現象・原因・対処法の順で説明する

#### 参考資料

- タイトル: URL

不明な点・調査できなかった点は必ず末尾に **`不明: ` で始まる行**で記載すること（例: `不明: ○○に関する情報が見つかりませんでした`）。この形式以外（「わかりませんでした」「情報なし」など）は使わない。補完・推測での記述は禁止。

---

## Step 3: 情報の取捨選択と統合

各エージェントから返ってきた調査結果を統合し、以下の基準で内容を整理する:

**残す情報**:

- 実運用上の課題と対応方法
- Issue の背景・目的と関連する情報
- 参考資料の URL

**除外する情報**:

- 詳細な理論・数式
- Issue の背景と無関係な情報

エージェントの出力に `不明: ` で始まる行が含まれているかチェックする。

- **不明行がない場合**: Step 4 へ進む
- **不明行がある場合**: 先に判明した調査結果を Step 4 でコメント投稿し、その後 **Step 3a** を実行する

### Step 3a: 不明点が残った場合

判明した分の調査結果を Step 4 で投稿した後、続けて `gh issue comment {番号} --repo Ilekaede/saji-image-processing-web --body "..."` で不明点コメントを投稿して止まる。

ユーザーから追加情報を受け取ったら、`/research {番号}` を再実行して Step 1 から再開する。

```
## 調査中に確認が必要な点（#<Issue番号>）

調査を行いましたが、以下の点について情報が不足しています。

- 不明: （不明点の内容）

ご確認後に情報をいただければ `/research <Issue番号>` を再実行して調査を補完します。
```

---

## Step 4: Issue コメント投稿

`gh issue comment {番号} --repo Ilekaede/saji-image-processing-web --body "..."` で以下の形式のコメントを投稿する:

```
## 調査結果（#<Issue番号>）

### <キーワード1>

#### よくある課題と対応方法

- **<課題名>**: <現象>。<原因>。対処法: <対処法>。

#### 参考資料

- <タイトル>: <URL>

### <キーワード2>

...

---

内容を確認し、記事作成を進めてよい場合はその旨を **Claude Code のチャット画面** に返信してください。
問題があれば修正点もチャット画面でお知らせください。
```

---

## Step 5: ユーザー確認・案内

ユーザーから記事作成を進める許可が出たら（「OK」「進めて」「問題ない」など文脈で判断する）、以下を案内して終了する:

```
記事執筆を開始するには以下を実行してください:

/edit-article <記事ID>
```

記事 ID が Issue 本文に記載されている場合はその数値を使う（例: 「method8」や「記事ID: 8」なら `8`）。記載がない場合は「記事 ID をご確認のうえ実行してください」と伝える。
`/edit-article` の自動呼び出しはしない。
