## 前提情報

- 記事IDは `$ARGUMENTS` から取得する（例: `/edit-article 8` → ID は `8`）
- 取得URL: `https://articles-worker.a-sakuramotyo.workers.dev/articles/method{ID}`
- R2バケット: `b3semi-articles`、オブジェクトキー: `method{ID}.md`
- 書き戻しコマンド（`cloudflare/articles-worker` ディレクトリで実行）:
  ```
  wrangler r2 object put b3semi-articles/method{ID}.md --file /tmp/article_edit.md --remote
  ```

---

## Step 1: 記事取得

`WebFetch` で以下のURLから記事を取得する。

```
https://articles-worker.a-sakuramotyo.workers.dev/articles/method{ID}
```

取得した内容を `現在の記事` として保持する。

---

## Step 2: 編集ループ（最大3回）

`iteration = 1` から開始し、最大3回繰り返す。

### 2-1. 編集エージェント（fork）

以下の指示と `現在の記事` を渡して fresh エージェント（subagent_type: claude）を起動する。
fork は会話コンテキスト全体を引き継ぐためスキル指示を読んでパイプライン全体を実行してしまう。編集作業のみ委ねるために fresh を使うこと。

---

**編集エージェントへの指示:**

あなたは画像処理の技術記事を書くエンジニアです。
以下の記事を「その手法を実運用する際の課題」にフォーカスした記事に再編集してください。

**編集方針:**

- 理論的な説明は冒頭1〜2段落に留める（アルゴリズムの詳細説明は不要）
- 記事のメインは「実際に動かすとぶつかる課題とその対処法」とする
- 各課題は「現象 → 原因 → 対処法」の構成で書く
- 対象読者は「その手法を実装してみたが上手くいかない経験を持つ学生・エンジニア」

**課題の洗い出し方針:**

記事の内容を読んだ上で、その手法を実運用する際に遭遇しやすい課題を網羅的に洗い出すこと。
課題は「入力データの質」「アルゴリズムの仮定の破綻」「実装・性能」の3カテゴリを軸に考えると漏れが少ない。

例として、オプティカルフローの記事であれば以下のような課題が考えられる（参考）:

- テクスチャ不足・低コントラスト（RGB値が似通った領域では特徴点を検出できない）
- ノイズ・画質劣化（低解像度・圧縮アーティファクトによる偽フローベクトル）
- 開口問題（エッジに沿った動きは1方向にしか推定できない）
- 照明変化・輝度恒常性の破綻（フリッカーや影でアルゴリズムが誤動作する）
- 高速移動・大変位（LK法・Farnebackは小さな変位を前提とする）
- モーションブラー（高速移動物体でエッジが滲み信頼性が低下する）
- カメラ自体の動き（手ぶれで物体運動と背景運動が混在する）
- オクルージョン（追跡点が別の物体に隠れたとき誤った点に乗り換える）
- 長期追跡のドリフト（フレームをまたぐごとに誤差が蓄積する）
- ROIの絞り込み（画面全体に流すと意味のないフィールドになる）
- 密フローの計算コスト（Farnebackはリアルタイム処理には重い）

**フォーマット:**

- Markdown（`##` `###` を使った見出し構成）
- コードブロックが必要な箇所はPythonコードで示す（OpenCV使用）
- 各課題は読み飛ばせる独立した節として書く

フィードバックがある場合は末尾に `## 前回フィードバック` として追記されているので、それも反映すること。

編集後の記事全文のみを返すこと。

---

fork エージェントが返した記事を `現在の記事` として更新する。

### 2-2. 並列レビュー（3エージェントを同一メッセージ内で同時起動）

`現在の記事` を渡し、以下の3エージェントを**1つのメッセージで並列に**起動する。

---

**レビュアーA: 実用性・再現性**

あなたは画像処理の実装経験を持つエンジニアです。
以下の記事を「実際に手を動かしたとき再現・応用できるか」という観点でレビューしてください。

チェック項目:

- コード例は実際に動くか（例: OpenCV APIの使い方が正しいか）
- 対処法が具体的か（抽象的なアドバイスになっていないか）
- 各課題の「現象・原因・対処法」が揃っているか

指摘事項を以下の形式でリストアップする:

```
[重大度: 高/中/低] 課題名または節名 - 指摘内容
```

---

**レビュアーB: 読みやすさ・わかりやすさ**

あなたはテクニカルライターです。
以下の記事を「その手法を実装中の学生・エンジニアにとって読みやすいか」という観点でレビューしてください。

チェック項目:

- 専門用語の初出に説明があるか
- 説明の流れが自然か（唐突に難しい話が出てこないか）
- 各節が独立して読めるか

指摘事項を以下の形式でリストアップする:

```
[重大度: 高/中/低] 課題名または節名 - 指摘内容
```

---

**レビュアーC: 構成・論理展開**

あなたは技術記事の編集者です。
以下の記事を「全体の構成と論理的な流れ」という観点でレビューしてください。

チェック項目:

- 課題の並び順が自然か（入門的な課題から複雑なものへの流れ）
- 節の重複・抜けがないか
- 導入と結びが全体を通じて一貫しているか

指摘事項を以下の形式でリストアップする:

```
[重大度: 高/中/低] 課題名または節名 - 指摘内容
```

---

### 2-3. 品質判定とループ継続判断

3つのレビュー結果を統合し、以下の基準で判断する:

**終了条件（両方満たすこと）:**

- 高重大度の指摘が 0 件
- 中重大度の指摘が 3 件以下

終了条件を満たした場合 → Step 3 へ進む

終了条件を満たさなかった場合:

- 全指摘を統合したフィードバックを作成する
- `iteration` をインクリメントする
- `現在の記事` の末尾に `## 前回フィードバック` として統合フィードバックを追記する
- 3回目のイテレーションが終了した場合は指摘が残っていても Step 3 へ進む

---

## Step 3: R2への書き戻し

1. `現在の記事`（末尾の `## 前回フィードバック` 節は除去してから）を `Write` ツールで `/tmp/article_edit.md` に保存する

2. `Bash` で以下を実行する（作業ディレクトリは `cloudflare/articles-worker`）:

   ```bash
   wrangler r2 object put b3semi-articles/method{ID}.md --file /tmp/article_edit.md --remote
   ```

3. コマンドが成功したことを確認する

---

以下の内容を報告する:

- 各イテレーションで検出した高/中/低の指摘件数
- 最終的に残った指摘のサマリー（あれば）
- R2アップロード結果
