## 前提情報

- リポジトリは `Ilekaede/saji-image-processing-web`
- ユーザーから「push して」と指示された場合もこのスキルのフローに従う（push と PR 作成は一度に行う）
- **不明・判断不能な場合は妄想で補完しない**。チャット画面で質問して止まる

---

## Step 1: 事前チェック

以下をすべて確認する。1つでも満たさない場合はその内容を報告して止まる。

1. **ブランチ**: 現在のブランチが `main` ではないこと（`main` なら「feature ブランチを切ってから再実行してください」と案内する）
2. **作業ツリー**: `git status` がクリーンであること（未コミットの変更があれば一覧を提示し、コミットするか退避するかユーザーに確認する。勝手にコミットしない）
3. **Issue 番号**: `$ARGUMENTS` に指定があればそれを使う。なければブランチ名 `feature/<番号>_<概要>` から取得する。どちらからも取得できない場合はユーザーに質問する

Issue 番号が確定したら `gh issue view <番号> --json title,body` で Issue を取得し、**背景・目的**を把握する（Step 4 の「背景」の材料にする）。

## Step 2: rebase

push 前に main の最新から生えている状態にする。

```bash
git fetch origin && git rebase origin/main
```

**コンフリクトした場合**は自動解消を試みない:

1. `git diff --name-only --diff-filter=U` でコンフリクトしたファイル一覧を取得する
2. `git rebase --abort` で rebase 前の状態に戻す
3. ファイル一覧とコンフリクトの概要を報告して止まり、ユーザーの指示を待つ

## Step 3: push

```bash
git push -u origin HEAD
```

rebase によって履歴が書き換わった既存ブランチ（過去に push 済み）の場合のみ `--force-with-lease` を付ける。それ以外で `--force` 系は使わない。

## Step 4: PR 本文の生成

`git log origin/main..HEAD --oneline` と `git diff origin/main..HEAD --stat` で変更全体を把握し、以下のフォーマットで本文を組み立てる。

### フォーマット（必須構成）

```markdown
closes #<Issue番号>

## 概要

### 背景

<なぜこの変更をするのか>

### 変更内容

- <変更点1>
- <変更点2>

## 動作確認

- [ ] <確認項目1>
- [ ] <確認項目2>

## レビュー観点

- <観点1>
- <観点2>
```

### 粒度ルール

各セクションは以下の粒度で書く。**変更規模によらずこの粒度を守る**こと。

| セクション   | 粒度                                                                                                                                                                                                     |
| ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 背景         | **1〜3行**。Issue の目的を要約し、PR 単体で「なぜやったか」が読める状態にする。調査過程・試行錯誤の経緯は書かない（必要なら Issue コメントに書く）                                                       |
| 変更内容     | **変更1件につき1行**の箇条書き。ファイルパス・関数名・コマンドはバッククォートで示す。目安は 3〜8 行。超える場合は `**カテゴリ**:` プレフィックスか小見出しで分類する。diff を逐語的になぞる説明はしない |
| 動作確認     | レビュワーが**コピペ・クリックで再現できる**チェックリスト。各項目に確認手段（コマンド・URL・操作手順）と期待結果を含める。「動くこと」だけの項目は不可                                                  |
| レビュー観点 | **2〜4項目**。重点的に見てほしい箇所を書く: ①迷った・自信のない設計判断とその理由 ②影響が波及しうる箇所 ③機能以外の評価軸（DX: Network タブ・コンソールの状態など）。「全体を見てください」は書かない    |

### タイトル

コミット規約と同じ形式で **70 文字以内**:

```
<emoji> <type>: <概要（日本語）>
```

## Step 5: PR 作成

```bash
gh pr create --title "<タイトル>" --body "<Step 4 の本文>"
```

作成後、PR の URL をユーザーに報告して終了する。`/code-review` の実行を提案してもよいが、自動実行はしない。
