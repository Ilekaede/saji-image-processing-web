## 前提情報

- 記事IDは `$ARGUMENTS` から取得する（例: `/edit-article 8` → ID は `8`）
- 取得URL: `https://articles-worker.a-sakuramotyo.workers.dev/articles/method{ID}`
- R2バケット: `b3semi-articles`、オブジェクトキー: `method{ID}.md`
- 作業ファイル（記事本文とフィードバックは必ずファイル経由で受け渡す）:

| ファイル                       | 役割                                                             |
| ------------------------------ | ---------------------------------------------------------------- |
| `/tmp/article_edit.md`         | 記事本文の唯一の置き場。編集エージェントが直接読み書きする       |
| `/tmp/article_feedback.md`     | 親が統合したレビューフィードバック（イテレーションごとに上書き） |
| `/tmp/edit_article_metrics.md` | ステージごとのトークン消費の計測ログ                             |
| `/tmp/research_notes.md`       | `/research` による事前調査結果（存在する場合のみ）               |

- 記事テンプレート: `.claude/skills/edit-article/article-template.md`（記事の構成・執筆ルールを定義。編集エージェントとレビュアーCが参照する）

---

## オーケストレーション必須ルール

トークン消費を抑えるためのルール。**どのモデルがオーケストレーションする場合も必ず守ること。**

1. **記事本文を親（このスキルを実行している Claude）のコンテキストに載せない。**
   `Read` で開かない・`cat`/`curl` で表示しない・サブエージェントへのプロンプトに貼り付けない。
   本文の受け渡しはすべて `/tmp/article_edit.md` 経由で行う。
2. **サブエージェントは fresh（`subagent_type: claude`）で起動し、`model` を必ず指定する。**
   編集エージェント = `sonnet`、レビュアー = `haiku`。
   `fork` は会話コンテキスト全体（このスキル指示を含む）を引き継ぎ、パイプライン全体を再実行してしまうため禁止。
3. **サブエージェントに記事本文を返させない。**
   編集エージェントは変更サマリーのみ、レビュアーは指摘リストのみを最終メッセージとして返す。
4. **ループの継続・終了判断は常に親が行う。**
5. **各ステージ完了時に計測ログへ1行追記する**（コマンドは各 Step に記載）。

---

## Step 1: 記事取得と計測ログ初期化

`Bash` で以下を実行する（本文を標準出力に表示しないこと）。

```bash
curl -sf -o /tmp/article_edit.md https://articles-worker.a-sakuramotyo.workers.dev/articles/method{ID}
printf '# edit-article 計測ログ (method{ID})\n\n| 時刻 | iteration | ステージ | 記事文字数 | 備考 |\n| --- | --- | --- | --- | --- |\n' > /tmp/edit_article_metrics.md
echo "| $(date +%H:%M:%S) | 0 | 取得直後 | $(wc -m < /tmp/article_edit.md) | - |" >> /tmp/edit_article_metrics.md
```

- 記事文字数はそのまま推定トークン数として扱う（日本語主体のため 1文字 ≒ 1トークンで換算）
- curl が失敗した場合や文字数が 0 の場合は処理を止め、ユーザーに報告する
- 前回実行の `/tmp/article_feedback.md` が残っている場合は `rm -f` で削除しておく

---

## Step 2: 編集ループ（最大3回）

`iteration = 1` から開始し、最大3回繰り返す。

### 2-1. 編集エージェント（fresh / model: sonnet）

`Agent` ツールを `subagent_type: claude`、`model: sonnet` で起動する。
プロンプトには以下の指示のみを渡し、**記事本文は含めない**。

---

**編集エージェントへの指示:**

あなたは画像処理の技術記事を書くエンジニアです。
`/tmp/article_edit.md` にある記事を「その手法を実運用する際の課題」にフォーカスした記事に再編集してください。

**作業手順:**

1. `Read` で `.claude/skills/edit-article/article-template.md` を読み込む（作業ディレクトリはリポジトリルート）（記事の構成・執筆ルール・セルフチェックリストを定義したテンプレート。**記事はこのテンプレートに従って書くこと**）
2. `Read` で `/tmp/article_edit.md` を読み込む（これが編集対象の記事）
3. `/tmp/research_notes.md` が存在すれば `Read` で読み込み、記載されている課題・対応方法・参考資料を課題の補強材料として優先的に活用する（存在しなければ下記「課題の洗い出し方針」に従って自身の知識で洗い出す）
4. `/tmp/article_feedback.md` が存在すれば `Read` で読み込み、前回レビューの指摘を必ず反映する
5. 編集後、テンプレート末尾のセルフチェックリストを確認する
6. 編集後の記事**全文**を `Write` で `/tmp/article_edit.md` に上書き保存する
7. 最終メッセージには**記事本文を含めず**、変更点のサマリー（箇条書き10行以内）のみを返す

**編集方針:**

- 記事の構成・フォーマット・文体はすべてテンプレート（article-template.md）に従う
- 記事のメインは「実際に動かすとぶつかる課題とその対処法」とする

**課題の洗い出し方針:**

記事の内容を読んだ上で、その手法を実運用する際に遭遇しやすい課題を網羅的に洗い出すこと。
課題は「入力データの質」「アルゴリズムの仮定の破綻」「実装・性能」の3カテゴリを軸に考えると漏れが少ない。

例として、オプティカルフローの記事であれば以下のような課題が考えられる（参考）:

- テクスチャ不足・低コントラスト（RGB値が似通った領域では特徴点を検出できない）
- ノイズ・画質劣化（低解像度・圧縮アーティファクトによる偽フローベクトル）
- 開口問題（エッジに沿った動きは1方向にしか推定できない）
- 照明変化・輝度恒常性の破綻（フリッカーや影でアルゴリズムが誤動作する）
- 高速移動・大変位（LK法・Farnebackは小さな変位を前提とする）
- モーションブラー（高速移動物体でエッジが滲み信頼性が低下する）
- カメラ自体の動き（手ぶれで物体運動と背景運動が混在する）
- オクルージョン（追跡点が別の物体に隠れたとき誤った点に乗り換える）
- 長期追跡のドリフト（フレームをまたぐごとに誤差が蓄積する）
- ROIの絞り込み（画面全体に流すと意味のないフィールドになる）
- 密フローの計算コスト（Farnebackはリアルタイム処理には重い）

（Markdown の見出し構成・コードブロックの書式・節の独立性など、フォーマットに関するルールはテンプレート article-template.md に従う）

---

編集エージェントの完了後、`Bash` で計測ログに追記する:

```bash
echo "| $(date +%H:%M:%S) | {iteration} | 編集後 | $(wc -m < /tmp/article_edit.md) | - |" >> /tmp/edit_article_metrics.md
```

### 2-2. 並列レビュー（fresh / model: haiku、3体を1メッセージで同時起動）

以下の3エージェントを `subagent_type: claude`、`model: haiku` で、**1つのメッセージ内で並列に**起動する。
プロンプトには観点別の指示のみを渡し、**記事本文は含めない**。

**3エージェント共通の指示（各プロンプトの冒頭に付ける):**

`Read` で `/tmp/article_edit.md` を読み込み、以下の観点でレビューしてください。
最終メッセージには指摘リストのみを返すこと（記事本文の再掲は不可、引用は1指摘につき1行まで）。
指摘は重大度の高い順に最大10件、以下の形式でリストアップする。指摘がなければ「指摘なし」とだけ返す。

```
[重大度: 高/中/低] 課題名または節名 - 指摘内容
```

---

**レビュアーA: 実用性・再現性**

あなたは画像処理の実装経験を持つエンジニアです。
「実際に手を動かしたとき再現・応用できるか」という観点でレビューしてください。

チェック項目:

- コード例は実際に動くか（例: OpenCV APIの使い方が正しいか）
- 対処法が具体的か（抽象的なアドバイスになっていないか）
- 各課題の「現象・原因・対処法」が揃っているか

---

**レビュアーB: 読みやすさ・わかりやすさ**

あなたはテクニカルライターです。
「その手法を実装中の学生・エンジニアにとって読みやすいか」という観点でレビューしてください。

チェック項目:

- 専門用語の初出に説明があるか
- 説明の流れが自然か（唐突に難しい話が出てこないか）
- 各節が独立して読めるか

---

**レビュアーC: 構成・論理展開・テンプレート準拠**

あなたは技術記事の編集者です。
「全体の構成と論理的な流れ」および「記事テンプレートへの準拠」という観点でレビューしてください。
レビューの前に `Read` で `.claude/skills/edit-article/article-template.md` を読み込み、末尾のセルフチェックリストを判断基準に使うこと（作業ディレクトリはリポジトリルート）。

チェック項目:

- 課題の並び順が自然か（入門的な課題から複雑なものへの流れ）
- 節の重複・抜けがないか
- 導入と結びが全体を通じて一貫しているか
- テンプレート準拠: 導入が2段落以内か、冒頭にベースコードがあるか、課題番号が通し番号か、まとめに逆引き表と「最初の一手」があるか

---

3体のレビュー完了後、`Bash` で計測ログに追記する（X/Y/Z は重大度別の指摘件数）:

```bash
echo "| $(date +%H:%M:%S) | {iteration} | レビュー後 | $(wc -m < /tmp/article_edit.md) | 高X 中Y 低Z |" >> /tmp/edit_article_metrics.md
```

### 2-3. 品質判定とループ継続判断

3つのレビュー結果を統合し、以下の基準で親が判断する:

**終了条件（両方満たすこと）:**

- 高重大度の指摘が 0 件
- 中重大度の指摘が 3 件以下

終了条件を満たした場合 → Step 3 へ進む

終了条件を満たさなかった場合:

- 全指摘を統合したフィードバックを `Write` で `/tmp/article_feedback.md` に保存する（上書き）
- `iteration` をインクリメントして 2-1 へ戻る
- 3回目のイテレーションが終了した場合は指摘が残っていても Step 3 へ進む

---

## Step 3: R2への書き戻し

記事本文はすでに `/tmp/article_edit.md` にあるため、そのまま `Bash` で以下を実行する（作業ディレクトリは `cloudflare/articles-worker`）:

```bash
wrangler r2 object put b3semi-articles/method{ID}.md --file /tmp/article_edit.md --remote
```

コマンドが成功したことを確認し、計測ログに追記する:

```bash
echo "| $(date +%H:%M:%S) | - | アップロード完了 | $(wc -m < /tmp/article_edit.md) | - |" >> /tmp/edit_article_metrics.md
```

---

## 完了報告

以下の内容を報告する:

- 実施したイテレーション数
- 各イテレーションで検出した高/中/低の指摘件数
- 最終的に残った指摘のサマリー（あれば）
- R2アップロード結果
- `/tmp/edit_article_metrics.md` の計測ログ（テーブルをそのまま提示し、ステージ間の文字数＝推定トークンの変化に触れる）
