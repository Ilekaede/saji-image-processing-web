# 記事テンプレート（実運用課題型）

`/edit-article` で記事を執筆・再編集する際は、必ずこのテンプレートに従うこと。
既存記事（method8 / method13 / method14）の分析から、読者が「手を動かして詰まったとき」に最も役立った構成をまとめたもの。

## 対象読者・ゴール

- 読者: その手法を実装してみたが上手くいかない経験を持つ学生・エンジニア（画像処理の初学者）
- ゴール: 読者が自分の症状から該当する課題節を逆引きし、コードをコピペして対処を試せること
- 前提: 実世界の画像はノイズ・オクルージョン・照明変化・天候などで理想状態にならない。理想論ではなく「実際に動かしたらこうなった」を書く

## 記事の構成（この順で書く）

### 1. タイトル

「〜の実運用で詰まる課題と対処法」「〜実装で詰まるポイント集」のように、**実装時の課題を扱う記事だとタイトルだけで分かる**形にする。

### 2. 導入（1〜2段落まで）

- 手法の一言説明（何を入力に何を出すか）
- その手法が置いている**基本仮定**を1つだけ挙げる（例: 輝度恒常性、等速直線運動）。後の課題節の多くはこの仮定の破綻として説明できるため、ここで押さえておくと記事全体が繋がる
- 「実装は数行で済むが、動かすと教科書通りにいかない」という橋渡しで本文へ
- 詳細な数式・アルゴリズム解説は書かない（他の資料に譲る旨を一言添えてよい）

### 3. 基本構成（ベースコード）

**冒頭に、最小構成で動く Python + OpenCV のコード全文を置く。**
以降の課題節のコードはすべてこのベースコードへの差分・追記として書く。
これにより各節のコード断片が「どこに足すのか分からない」状態になるのを防ぐ（method13 で最も効果があった形式）。

- 動画読み込み〜メインループ〜後始末まで含めた、コピペで動く完結したコードにする
- 外部依存（検出器など）が必要な場合は `detect(frame)  # 検出器（別途実装）` のように明示する

### 4. 課題節（記事のメイン）

各課題を以下の型で書く:

```markdown
### 課題N: <読者が体験する症状をそのまま書いた見出し>

**現象**
実行するとどう見えるか。読者が「自分のことだ」と気づける具体的な描写。

**原因**
なぜ起きるか。アルゴリズムの仮定・パラメータ・入力データのどれが破綻しているか。

**対処法**
何をどう変えるか。コード付き。
```

**ルール:**

- 課題番号は記事全体の**通し番号**にする（セクションごとに 1 からリセットしない。method14 の「課題1が2回出てくる」問題を避ける）
- 課題数が 8 個を超える場合は「入力データの質」「アルゴリズムの仮定の破綻」「実装・性能」の3カテゴリで `##` 見出しを立てて分類する（番号はカテゴリをまたいで通しのまま）
- 各節は独立して読める状態にする。別の節の実装が前提になる場合は、節の冒頭に「> この節は課題Nの実装が前提です」と明記する
- 見出しは「開口問題」のような用語ではなく「エッジ上の追跡が不安定になる（開口問題）」のように**症状ファースト**で書く

### 5. まとめ

- 課題 × 主な対処法の一覧表（読者が後から逆引きするための索引）
- 「まずどこから手を付けるか」の指針を1段落で書く（例: どの課題が支配的かの特定方法、シンプルな手法から段階的に重ねる順序）
- 複数の手法を扱う記事では「状況 → 向いている手法」の選択指針表も付ける（method14 の形式）

## 執筆ルール

### コード

- Python + OpenCV。ベースコードへの差分として、単体でコピペして動く状態にする
- 未定義の変数を突然使わない（例: `flow_history` をどこで作るのかを示す）。ループ文脈が必要なコードはループごと書く
- 概念だけ示す擬似コードには `# 概念コード:` とコメントで明示する

### パラメータ

- パラメータを出すときは**意味・目安値・決め方**をセットで書く（例: `varThreshold` は前景判定の閾値、デフォルト16、動的背景では40程度に上げる）
- 目安値には根拠を付ける。実測手順（グラウンドトゥルースとの誤差計測など）や逆算式（目標 fps → 1フレーム 33ms → 最大粒子数）を示せるとよい

### トレードオフと現実的な選択肢

- 対処法にトレードオフがあれば必ず明示する（例: ブラーを強くするとノイズは消えるが細かい動きも潰れる。感度を上げる対処と下げる対処が逆方向で衝突する場合はその旨を書く）
- アルゴリズム側の対処だけでなく、**撮影条件・ハードウェア側の対処**も選択肢に入れる（シャッタースピード、フレームレート、カメラの設置角度、ジンバルなど）。「コードで頑張るよりカメラ位置の見直しが安い」ことは現場では珍しくない

### 文章

- 「です/ます」調。丁寧だが堅すぎない
- 専門用語の初出には括弧書きで短い説明を付ける（例: CLAHE（コントラスト制限付き適応ヒストグラム均等化））
- 一文を長くしすぎない。導入段落で3行を超える一文は分割する

## セルフチェックリスト

執筆・編集後に以下を確認する:

- [ ] 導入は2段落以内で、数式・理論の詳細に踏み込んでいない
- [ ] ベースコード（動く最小構成）が冒頭にある
- [ ] すべての課題節に「現象・原因・対処法」が揃っている
- [ ] 課題番号が通し番号になっている
- [ ] コード例に未定義変数がない（あるいは参照先の節を明示している）
- [ ] 主要パラメータに意味と目安値・決め方が書かれている
- [ ] まとめに逆引き表と「最初の一手」の指針がある
