## 前提情報

- リポジトリは `Ilekaede/saji-image-processing-web`（`gh` コマンドには必ず `--repo` を付ける）
- 検証方式は #142 の調査で決定済み: **AI が PR の diff を読んで検証計画を組み立て、ブラウザを操作して検証する**（テストコードを積み上げる方式ではない）。回帰保証にはならない
- 結果報告は `/code-review` と同じ流儀: `--comment` 付きなら PR コメントに投稿、なければチャットに報告して投稿を提案する
- **不明・判断不能な場合は妄想で補完しない**。チャット画面で質問して止まる

### できる / できないの地図（検証計画の判断基準）

| 検証したいこと                                 | 可否 | 手段                                       |
| ---------------------------------------------- | ---- | ------------------------------------------ |
| 要素の表示・クリック遷移                       | ◎    | スクリプト or MCP                          |
| フェッチの回数・ステータス（二重呼び出し検証） | ◎    | スクリプト（ネットワーク傍受）             |
| コンソールエラーの有無                         | ◎    | スクリプト                                 |
| レイアウトが大きく崩れていないか               | ○    | スクショを AI が目視                       |
| ピクセル単位の見た目比較                       | △    | やらない（framer-motion で誤検知しやすい） |
| アニメーション・動画の動きの正しさ             | ✕    | 検証対象外としてレポートに明記             |
| OpenCV デモの処理結果の数値的な正しさ          | ✕    | 検証対象外としてレポートに明記             |
| デザインの良し悪し・読みやすさ                 | ✕    | 検証対象外としてレポートに明記             |

### ブラウザ操作の使い分け

- **目視・探索系**（要素を探す・画面の状態を対話的に確かめる・セレクタの当たりをつける）→ **Playwright MCP**
- **計測系**(フェッチ回数・ステータス・コンソールエラー数・スクショの一括取得) → **使い捨てスクリプト**（機械的に正確）

## Step 1: PR の把握と検証計画

1. PR 番号を確定する: `$ARGUMENTS` に指定があればそれを使う。なければ `gh pr view --json number,title,headRefName` で現在のブランチの PR を取得する。どちらでも取得できなければユーザーに質問して止まる
2. **検証対象のブランチがチェックアウトされていることを確認する**。現在のブランチが PR の head と異なる場合は、`git status` がクリーンであることを確認したうえで `gh pr checkout <番号>` する（未コミットの変更があれば報告して止まる）
3. `gh pr view <番号> --json title,body` と `gh pr diff <番号>` で変更内容を読む
4. 「できる / できないの地図」に従って**検証計画**を組み立てる。変更の種類ごとの典型例:
   - **記事・カード追加**: カードの表示 → View More クリックで遷移 → 記事本文の描画（見出し・コードブロック・本文量）→ フェッチ回数 → スクショ目視
   - **UI 変更**: 対象ページの表示 → 変更された要素の存在・状態 → スクショ目視
   - **データ取得まわり**: フェッチの回数・ステータス・URL → コンソールエラー
   - すべての PR で共通: **コンソールエラー 0 件**の確認を必ず含める
5. 計画をチャットに提示してから実行に進む。**「何を検証するか」と「何を検証対象外とするか（✕ 項目）」の両方を明記する**

## Step 2: dev サーバ起動

1. `curl -sf -o /dev/null http://localhost:3000/saji-image-processing-web/` で既に起動していればそれを使う
2. 起動していなければ `pnpm dev` をバックグラウンドで起動する。**port 3000 固定**（vite.config.ts が `strictPort` のため、衝突時は失敗する。articles-worker の CORS 許可リストの都合で別ポートへの回避は不可 → 3000 を塞いでいるプロセスを特定して報告し、止まる）
3. ベース URL は `http://localhost:3000/saji-image-processing-web`（`basename` があるためパス必須）

## Step 3: 検証の実行

1. `templates/verify-template.cjs` を scratchpad にコピーし、「検証本体」セクションを Step 1 の計画に合わせて書き換える。収集系（コンソール・ネットワーク・スクショ）のコードは書き換えない
2. 実行する:

   ```bash
   SCREENSHOT_DIR=<scratchpad> NODE_PATH=<プロジェクトルート>/node_modules node <スクリプト>.cjs
   ```

   `browserType.launch: Executable doesn't exist` で落ちた場合は `pnpm exec playwright install chromium` を実行してからリトライする

3. **セレクタの想定違いで 2 回失敗したら、スクリプトを直し続けずに Playwright MCP で実画面を開いて構造を確認してから書き直す**（PoC の教訓。例: カードタイトルはクリック不可で View More ボタンが遷移担当・記事 URL は小文字 `/methods/{id}`）
4. スクリプトの末尾に出る `SUMMARY(JSON)` を検証結果の一次ソースとする

## Step 4: スクショの AI 目視

1. スクリプトが保存したスクショを Read で開いて目視する
2. 観点は「レイアウトが大きく崩れていないか・要素が欠けていないか・明らかな異常がないか」に限定する。ピクセル単位の差分やデザインの良し悪しは判断しない
3. fullPage スクショに半透明の要素が写っている場合は framer-motion のスクロールイン演出の写り込みを疑う（テンプレートの `scrollFullPage` を通してから撮り直す）

## Step 5: レポート

以下のフォーマットでレポートを組み立てる:

```markdown
## MR 検証結果（#<PR番号>）

`/verify-pr` によるローカルブラウザ検証（回帰保証ではありません）

### 検証結果

| 検証項目     | 手段           | 結果      |
| ------------ | -------------- | --------- |
| <検証項目1>  | スクリプト     | ✅ <補足> |
| <検証項目2>  | Playwright MCP | ✅        |
| スクショ目視 | AI 目視        | ✅ <所見> |

### 検証対象外

- <✕ 項目のうち、この PR に関係するもの>（理由: <地図の理由>）

### 気づいた点

- <検証中に見つけた計画外の問題。なければセクションごと省略>
```

- `--comment` が指定されている場合: `gh pr comment <番号> --repo Ilekaede/saji-image-processing-web --body "..."` で投稿し、チャットには要約と投稿 URL を報告する
- 指定がない場合: チャットにレポート全文を報告し、PR コメントへの投稿（`--comment` 相当）を提案する。自動投稿はしない
- ❌ があった場合は結果をそのまま報告する（勝手に修正しない。修正はユーザーの指示を待つ）
- このスキルが起動した dev サーバは最後に停止する（元から起動していたものは触らない）
